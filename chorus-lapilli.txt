Chorus Lapilli Implementation:

First thing I decided to tackle was the difference between traditional tic-tac-toe and this version, the switch between placing pieces and moving them(after both players have moved 3 times). I started by adding a function to board that finds the number of Xs and Os on the board, and returns handleClick early if that number for whichever one is up next exceeds 3(effectively stopping placement).

Noticed a duplicate if statement call(using xIsNext twice to decide whether character was an X or O), so I refactored the rest of the function to prevent any duplicate if statements(if statement already needed to be called before for player). At this point, on the user side it looks as expected, where no more moves were possible after 3 X's were put down. 

Then I started to tackle the moving portion, where the first click has to select and the second has to place it. Started with implementing a new var startSquare(via useState, board needs to remember it), and fed it into handleClick. Then, I started the moving portion, where I refactored my conditional so that it first handled the placement phase, then started handling the movement phase.

For the movement phase, I started by handling the startSquare portion first(startSquare is null), with an extra check to make sure the player selected a square that was their own. Then, I handled the destination Square portion, which checked whether the new square was empty and adjacent to the startSquare(using a helper function and array of arrays, otherwise readability would have suffered), and updates the squares if both checks were satisfied, then resets startSquare. After this was implemented, I tested it out, and it worked as intended(first click selected a square, second), except for a small issue(whenever jumpTo was used, the code kept the initial one selected, instead of reseting, causing 4 tiles of one type to show up if an empty tile was selected after).

To fix that issue, I relocated startSquare to Game(so that i could be tracked across jumps, so I could reset selection in jumpTo), and passed it into Board and Square via props. This resolved the missing refresh issue, where the program functioned as intended on the user side.

Last, I decided to tackle the Center Rule, which I implemented through the helper function CenterRule for readability. As the center rule only applies for the moving portion and the adjacency check already exists, the function itself did not have to check for 3 pieces on board or that the start and end square were the same, so the function checked for the remaining two criteria(next move must either win or vacate square) via series of checks(checking if the middle square is empty, if it was already selected as the start square, or if the current possible move would win the game). 

After implementation, checking on the localhost showed that the center rule was functioning as expected, with no errors.
